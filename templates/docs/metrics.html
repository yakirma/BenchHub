Writing metrics

A metric is a small Python function that scores a prediction against ground truth. Author it once, reuse it across leaderboards.

The shape of a metric

A metric is plain Python source stored in the metric library. It receives one argument per mapped field and returns a number. numpy (as np) and benchhub (as bh, also benchhub) are pre-injected — usable without an import. Anything else from the standard library (math, json, …) you import inside the function body.

def rmse_depth(gt, pred):
    import numpy as np
    return float(np.sqrt(np.mean((np.asarray(gt) - np.asarray(pred)) ** 2)))

Per-sample vs aggregated

Per-sample (the common case)

Returns one number for one sample. BenchHub runs it on every sample and pools the results into the leaderboard score (mean / median / percentile / min / max — chosen on the binding). Each per-sample value is also stored, so re-pooling is instant.

Aggregated

Receives the lists of values across all samples (e.g. all GT labels + all predicted labels) and returns one number for the whole submission — for metrics like AUROC or a corpus-level BLEU that can't be averaged per sample.

Typed inputs (recommended)

Annotate the arguments with the kind you expect, and BenchHub hands you decoded bh.<Kind> instances instead of raw primitives. The annotations are parsed at save time into the metric's accepted kinds, and the argument names map to roles by convention (gt, pred, input).

import benchhub as bh

def accuracy(gt: bh.Label, pred: bh.Label | bh.LabelList):
    # gt.value is the true class index; pred.value (or pred.values[0]) the prediction
    p = pred.value if isinstance(pred, bh.Label) else pred.values[0]
    return 1.0 if p == gt.value else 0.0

Use | unions to accept multiple kinds. Put an isinstance check in the body if you branch on type. Metrics with no annotations keep receiving the raw primitive value (back-compat).

Built-in reference metrics

Five typed metrics ship ready to bind:

  • accuracy — classification (bh.Label / bh.LabelList);
  • rmse_depth, mae_depth — depth error (bh.Depth);
  • iou_mask — segmentation IoU (bh.Mask);
  • exact_match_text — string equality (bh.Text).

Binding a metric to a leaderboard

Authoring a metric just adds it to the library; it scores nothing until you bind it to a leaderboard (on the leaderboard's Edit page). The binding sets:

  • argument mappings — each function arg → a dataset or prediction field (e.g. gt→label, pred→label_pred);
  • display name — the column header (and the handle other metrics chain on);
  • pooling, sort direction, and an optional tag filter.

Chaining metrics

A metric can consume another metric's per-sample output by mapping an argument to that metric's display name. BenchHub topologically sorts the bound metrics so dependencies run first, then feeds the upstream values in. This lets you build, say, a normalized score on top of a raw error without recomputing the error.

Authoring & managing

  • Write and edit metrics on the Metrics page. Each metric is Python source you can overwrite in place — no migration.
  • Visibility: your metrics default to private; publish to public to let others bind them. Names are unique per owner, and globally unique only among public metrics (so two people can each have a private my_iou).
  • Sandboxing: metric code is executed server-side. Deployments can set BENCHHUB_SANDBOX_METRICS=1 to run it in an isolated sandbox.
A metric is just code — spot-check anything non-trivial (rank-based, span-overlap, BLEU-family) on a couple of samples in the comparison view before trusting the column.
