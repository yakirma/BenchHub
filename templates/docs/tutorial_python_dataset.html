Build a dataset in Python

Have data already in memory or on disk? Skip the importer and push a typed dataset straight from Python with the client — no HuggingFace repo, no file tree, no commit-time mapping. You declare the schema and stream the samples yourself.

The importer exists to fill a typed contract from a repo that already lives somewhere. But the contract is the spine, not the repo — if you already hold the bytes (a generated eval set, a CSV plus a folder of images, the output of yesterday's data pipeline), there's nothing to import. The client lets you build the same typed dataset directly: declare each field's kind and role, then hand it typed values sample by sample. Everything downstream — decode, render, score, the comparison view — works identically to an imported dataset, because the contract is identical.

When to use this. Reach for the client when the data isn't on HuggingFace in a clean form, or isn't worth round-tripping through a repo: synthetic / generated evals, a private slice you can't publish, anything you can already load into Python. Reach for the importer instead when the data already sits in an HF (or Kaggle) repo — one button beats re-uploading bytes you don't own.

The four moves, in code

Same shape as every other tutorial — bring data in → declare the contract → bind a metric → submit & compare. Here the first two moves happen in Python:

import benchhub as bh
client = bh.Client()                       # reads $BENCHHUB_API_TOKEN

ds = client.create_dataset("my-eval", visibility="private")
ds.add_field("image", kind="image", role="input")
ds.add_field("answer", kind="text",  role="gt")
for name, img, ans in load_my_data():
    ds.add_sample(name, image=bh.Image(img), answer=bh.Text(ans))
print(ds.create())

load_my_data() is your own loader (pseudocode here) — whatever yields your (name, image, answer) tuples, from a folder, a CSV, or a pipeline in memory.

That's the whole dataset. The leaderboard, metric, and submission steps are then identical to an imported dataset (covered below). New to kind, role, or contract? See Core Concepts; the full list of kinds and their constructors is in Data Types.

Step by step

1 · Connect the client

Install once (pip install benchhub-client), then construct a client. It reads your API token from the environment, so nothing secret lives in the script:

import benchhub as bh
client = bh.Client()                       # reads $BENCHHUB_API_TOKEN
# or pass it explicitly:
# client = bh.Client(token="<your-api-token>")

Generate a token at Settings → API tokens (also reachable from Settings → Account); this requires a signed-in account. The base URL defaults to https://runbenchhub.com; override it with $BENCHHUB_BASE_URL if you run your own instance.

2 · Create the dataset

ds = client.create_dataset("my-eval", visibility="private")

create_dataset returns a local builder — nothing is uploaded yet. It defaults to public, so pass visibility="private" (as the example does) to keep the dataset yours alone until you choose to share it. (You can also flip visibility later from the dataset page.)

3 · Declare the schema with add_field

Each field declares a name, a kind (its declared type), and a role. The role is the load-bearing decision:

  • role="input" — conditioning given to the submitter. Shipped to the model at submit time; not a prediction target.
  • role="gt" — the ground truth (the held-aside answer). Kept server-side and scored against; never handed to the model.
ds.add_field("image",  kind="image", role="input")
ds.add_field("answer", kind="text",  role="gt")

Some kinds take params — pass them with params=. A label field's class vocabulary and a depth field's unit are the common ones:

ds.add_field("cls",   kind="label", role="gt",
             params={"names": ["cat", "dog", "bird"]})
ds.add_field("depth", kind="depth", role="gt",
             params={"unit": "meters"})
Roles aren't cosmetic — they decide the contract. When you build a leaderboard on this dataset, BenchHub derives the prediction contract from your gt fields (an answer GT → an answer_pred field submitters fill in) and ships the input fields to the model. Mislabel an answer column as input and it leaks to the model; mislabel a conditioning image as gt and the board asks people to predict it. Get the roles right here and the rest follows.

4 · Add samples with typed values

Stream your data through add_sample(name, **typed_values). The first argument is the sample's stable name (used to align GT with predictions and to navigate the comparison view); the keyword arguments match your declared field names, each wrapped in the matching bh.<Kind>(…) constructor:

for name, img, ans in load_my_data():
    ds.add_sample(name, image=bh.Image(img), answer=bh.Text(ans))

Wrapping the value in bh.Image(…) / bh.Text(…) / bh.Label(…) / bh.Depth(…) … is what makes the dataset typed: the kind travels with the bytes, so the platform never has to guess how to decode or render them. bh.Image takes a NumPy array (e.g. (H, W, 3) uint8); the array kinds (image, mask, depth) expose .array when read back. Use a name that is unique and stable across runs — re-using a name merges fields into that existing sample (overwriting any field you pass again) rather than adding a duplicate.

5 · Upload with ds.create()

print(ds.create())

create() packs everything you added and uploads it in one call — this is the first time bytes leave your machine. It returns the created dataset's identity, which you'll use to build a leaderboard. The dataset is stored as the lightweight preview tier (small thumbnail-resolution copies for browsing, not full-resolution bytes); when you later create a leaderboard that scores on a subset, only that subset's full-resolution bytes are materialised. Because the example passed visibility="private", it stays private until you choose to share it.

From dataset to scored leaderboard

You now have a dataset that is indistinguishable from an imported one — same typed contract, same preview tier, same visibility controls. So the remaining moves are exactly the canonical loop:

  1. Create a leaderboard. Open the dataset → New leaderboard. The prediction contract is derived from your gt fields (here: answer_pred, kind text); your input fields are what the model receives. Pick a sample selection and bind a metric, mapping its gt arg to answer and its pred arg to answer_pred.
  2. Submit. Iterate the board's samples, run your model, predict, submit — the same reusable loop as everywhere else.
  3. Compare. Open the comparison view to see GT vs each prediction, sample by sample.

The submission loop, for the contract above:

lb = client.leaderboard("you/my-eval-board")   # the board's "Client handle"
sub = lb.submission(name="my-model")
for name, inputs in lb.iter_samples():
    img = inputs["image"].array                # decoded bh.Image
    sub.predict(name, answer_pred=bh.Text(my_model(img)))
sub.submit()

Inputs arrive as decoded bh.<Kind> instances and are cached on disk after the first download, so re-runs are offline. The full submission walkthrough — reusable script, Colab, re-scoring — is in the first-submission tutorial, and the complete client API is in Submitting.

One contract, every entry point. Importing, the file-tree mapper, and the Python client all converge on the same typed dataset — once you've built one in code, everything you learned about leaderboards, metrics, and the comparison view applies unchanged.
