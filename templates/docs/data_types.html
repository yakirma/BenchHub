Data types (kinds)

Every field of data has a kind. The kind decides how a value is stored, decoded, scored, and rendered.

BenchHub's typed contract is defined once in code (benchhub/types.py) and used everywhere — the dataset, the benchhub-client, and the metric engine. The canonical, always-current list lives at /supported_types (rendered straight from the code); this page explains each kind with examples.

On the Python side each kind is a class in the benchhub package (import benchhub as bh). You build a value with the constructor (e.g. bh.Image(array)) and the client encodes it for you. When you read inputs back via iter_samples, you get these same instances.

Visual kinds

image — RGB(A) or grayscale photo

A uint8 array, shape (H,W), (H,W,3), or (H,W,4). Stored as PNG. Rendered as the image.

bh.Image(np.zeros((256, 256, 3), dtype=np.uint8))
mask — integer label map

An integer array (H,W) of class IDs. Stored as a single-channel PNG (8- or 16-bit). Rendered with a deterministic color palette. Params: num_classes, ignore_index (default 255).

bh.Mask(seg, num_classes=21, ignore_index=255)
depth — per-pixel depth map

A float array (H,W). Stored as NPZ. Rendered with a colormap (turbo by default); the comparison view lets you switch colormaps (turbo, viridis, magma, jet, gray, normal…) live. Param: unit (default "meters").

bh.Depth(depth_array, unit="meters")
sequence — a clip of frames (video)

An ordered list of homogeneous frames (all image, or all depth/mask) treated as one sample. Stored as a ZIP of the per-frame encodings; rendered as an mp4 video in the dataset and comparison columns. Params: item_kind (the frame kind) and fps. The value is iterable — for frame in clip: frame.array.

frames = [bh.Image(f) for f in my_clip]   # list of (H,W,3) uint8
bh.Sequence(frames, item_kind="image", fps=10)
bboxes — bounding boxes · coco_detections — COCO-style detections

Object boxes (with an optional format such as xyxy). bboxes renders as an SVG overlay; coco_detections carries COCO-shaped detection records for overlay visualizations.

audio — waveform

A samples array plus a sample_rate. Stored as WAV; the preview is a waveform PNG.

bh.Audio(samples, sample_rate=16000)

Label kinds

label — one class

A single class, stored inline. The value is an int index (or a string); an optional names vocabulary maps the index to a human-readable class name, so the UI shows cat rather than 3. Used as the target for classification metrics.

bh.Label(3, names=["dog", "cat", "bird", "fox"])   # -> "fox"
label_list — ranked top-K

An ordered list of class predictions (most-confident first), for Top-K / Hits@N / MRR metrics. Requires k; optional names vocabulary like label.

bh.LabelList([2, 0, 5], k=5, names=class_names)

Inline value kinds

scalar · text · json
  • scalar — a single float, stored inline. Good for precomputed scores or numeric attributes.
  • text — free-form string (captions, answers).
  • json — the escape hatch for structured data that doesn't fit another kind (nested dicts/lists, custom records).
bh.Scalar(0.92)
bh.Text("a small brown dog")
bh.Json({"answer": "Paris", "spans": [[3, 8]]})

How kinds are stored on disk

A typed dataset is a directory with a manifest.json plus one folder per field holding <sample>.<ext> for every sample. File-backed kinds (image/mask/depth/audio/sequence, and also text/json) live as files; only the truly inline kinds (scalar/label/label_list) ride in the manifest/database. (Text and JSON are additionally mirrored into the database for quick rendering, but on disk they're .txt/.json files.) You rarely build this by hand — the importers and the BHDatasetCreator produce it for you.

Picking the wrong kind is the most common import mistake (e.g. a packed .npz isn't always depth). The file-tree importer has a decode preview so you can confirm a field decodes correctly before committing.
