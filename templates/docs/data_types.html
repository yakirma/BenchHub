Supported data types

How BenchHub recognizes, stores, and serves the fields in your datasets and submissions.

BenchHub auto-detects the type of each field from a combination of folder prefix and file extension. The same rules apply whether the field lives on the ground-truth side (a Dataset uploaded as a ZIP or a HuggingFace dataset attachment) or on the prediction side (a Submission uploaded via the API). Every per-LB "Submission contract" you see on a leaderboard page is just a filtered view of these rules.

Folder names that don't start with a reserved prefix (metric_, raw_, hist_, raw_histogram) are still indexed — their type is decided by the file extension alone. The reserved prefixes carry extra semantics (see below).

Reference table

Type File ext. Per-sample naming How it's stored / read
scalar .txt {{ '.txt' }} UTF-8 text containing a single number. Most often a float (0.873) or an integer class index. Parsed to float when handed to metric code.
text .txt {{ '.txt' }} UTF-8 string. Free-form — generated captions, sentiment labels (pos/neg), short classifier outputs. Passed to metric code as a Python str.
image .png / .jpg / .jpeg / .bmp / .tiff {{ '.' }} RGB or grayscale raster. Read as a PIL.Image when a metric / visualization asks for it. The submit scaffold auto-converts numpy arrays / PIL Images to PNG HxWx3 uint8.
mask .png {{ '.png' }} Segmentation mask — either single-channel HxW (uint8/16 class IDs) or three-channel HxWx3 RGB-encoded classes. The engine inspects the shape and picks the right decode path.
depth .npz {{ '_x.npz' }} numpy .npz archive with one array under key depth (HxW float). The <W>x<H> in the filename lets the engine validate shape without loading. Folder names typically start with raw_.
histogram .npz {{ '.npz' }} numpy .npz archive with two arrays: bins + counts. Detected when the folder starts with hist_ or is named exactly raw_histogram. Surfaces in the Comparison view as an inline chart.
json .json {{ '.json' }} Arbitrary structured data, parsed to a Python dict. Useful for per-sample config or complex predicted objects (bboxes etc.).

Reserved folder prefixes

BenchHub recognizes three folder prefixes that carry semantics beyond the file extension:

metric_
Each <sample>.txt in the folder holds a precomputed metric value (one float per sample) — not a raw prediction. BenchHub reads the values directly and skips re-running the metric. Use this when you already evaluated locally and just want to ship the scores. Never use metric_ for raw model outputs — pair the metric with a bare-name prediction folder instead.
raw_
Marks the folder as carrying dense raw arrays — most commonly depth maps (raw_depth_map) and free-form numpy payloads. The <W>x<H> suffix on each filename gives the engine the spatial shape up front.
hist_
Marks the folder as histogram data — each .npz carries bins + counts arrays. Folder name suffix is the histogram variant (e.g. hist_pred, hist_smooth); the comparison view surfaces each variant as a separate chart.

Using data in metrics

Detected fields become arguments you map onto a metric function in the Leaderboard Settings. The dispatch table:

  • scalar / text → Python float / str passed by value.
  • image / mask → file path; load with PIL.Image.open(path).
  • depth → file path; load with np.load(path)['depth'].
  • histogram → pre-loaded dict with 'bins' + 'counts' arrays.
  • json → pre-loaded Python dict.

The per-LB "Submission contract" widget on every leaderboard page lists exactly which folders that LB requires — same rules, scoped to the metrics + visualizations on that LB.
