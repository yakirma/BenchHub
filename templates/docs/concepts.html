Core concepts

The handful of objects everything else is built from — and why each one is shaped the way it is.

Why a typed contract?

ML benchmarking is usually a pile of one-off glue: every benchmark invents its own data layout, its own scoring script, its own submission format. Results barely reproduce, are even harder to compare, and what you get back is a single number that tells you nothing about where your model actually fails.

BenchHub's answer is one idea: declare the data contract once, and make it typed. Every field — a dataset column, a leaderboard input, a prediction — carries a declared kind (image, mask, depth, audio, text, label, scalar, sequence, and more; you can register your own). Because the dataset, the client, and the metric engine all agree on the kind, the platform never has to guess: decode, render, colormap, score, and compare all fall out of that single decision. If both sides agree a field is a depth map in meters, it is handled the same way every time — across datasets, models, and metrics.

Everything downstream is a consequence — and each idea below gets its own section, so don't worry if these are new. The same typing lets BenchHub store a cheap browsable preview of every sample and pull full-resolution bytes only where a benchmark needs them (two-tier storage and materialization). It lets anyone contribute scoring code that runs safely on other people's data (the sandbox). And it lets the comparison view line up the correct answer against each model's prediction, sample by sample — turning the leaderboard from one number back into the inspection tool you came for. Read on; each is defined where it appears.

The rest of this page builds the object model smallest to largest, so by the end the rest of the docs name things you already understand. Each concept gets a one-line why where it's a deliberate design choice.

Kind

The atom. A kind is the declared type of a piece of data — image, mask, depth, audio, text, bboxes, label, label_list, scalar, json, sequence (an ordered clip of frames that renders as an mp4), coco_detections, and any custom types you register.

Why: the kind is the single decision the whole platform leans on. Decode, render, colormap, score, and compare are all derived from it — never guessed. The authoritative list lives on the Data Types page.

Field & roles

A field is a name + a kind + a role. The role says how the field participates in a benchmark:

RoleMeaning
inputGiven to the submitter at inference time (e.g. the RGB image). Not predicted.
gtGround truth (GT) — the correct answer the dataset ships with, held server-side as the target a prediction is scored against.
predA field the submitter produces. Declared on the leaderboard's prediction contract.

Why roles are set per leaderboard: so one dataset can be used many ways — the same image+depth pair can be a depth-estimation benchmark (image is input, depth is gt) or an image-generation benchmark elsewhere. Handy default: a lone image field becomes input and the rest become gt.

Sample

A sample is one row: a single value for each field. An image sample might hold an RGB image, its ground-truth depth map, and a label — one value per field, each carrying its declared kind.

Dataset

A dataset is a collection of samples and fields. It is global — not scoped to any user or project namespace, so any leaderboard can bind any dataset.

Datasets are stored in a two-tier way:

  • Preview tier (always present): a small, downscaled rendering of every sample — ≤512px JPG thumbnails for image/mask/depth, a waveform PNG for audio, the inline value for text/label/scalar/json. Tens of KB per sample. This is what you browse.
  • Materialized tier (per leaderboard, on demand): the full-resolution bytes for the subset of samples a leaderboard actually scores on. See Materialization below.

Why two tiers: browsing a dataset shouldn't cost as much as running it. The preview tier caches a whole dataset cheaply so you can explore every sample for tens of KB each; the heavy full-resolution bytes only land when a benchmark genuinely needs them — and then only for the samples it uses.

Leaderboard

A leaderboard binds one (or more) datasets and defines a benchmark:

  • Field roles — which fields are inputs vs ground truth.
  • A prediction contract — the list of fields a submitter must produce, each named <x>_pred with a kind.
  • Metrics and visualizations bound to it (see below).
  • A materialized sample subset (see below).

Why the prediction contract is frozen: it locks once the leaderboard has submissions, so existing predictions are never re-decoded through a different decoder. Changing a field's kind afterwards would re-interpret already-uploaded prediction files the wrong way — so the bytes you submitted always mean what they meant when you sent them.

Materialization

Browsing only needs the cheap preview tier, but scoring needs the real bytes. So when you create a leaderboard you pick a sample subset to materialize at full resolution:

  • Sample cap — how many samples (or all of them).
  • Sampling — head (first N), random, or stratified (balanced across a label field). Random is the default; stratified is auto-suggested when the dataset has a label field.
  • Seed — makes random/stratified picks reproducible.

Materialization runs in the background; the leaderboard page shows its status (pending → running → ready, or failed with a retry). Re-materializing is blocked once the leaderboard has submissions — changing the sample set would invalidate the inputs those submissions were scored against.

Why per leaderboard: full-res cost should be proportional to use. Each leaderboard pays for exactly the subset it scores on, charged to its owner's quota — not for the whole dataset, and not by everyone who browses it.

Submission

A submission is one model's set of predictions for a leaderboard. Each carries a processing status (Pending → Processing → Processed, or Error) and, once scored, a row of metric results. You submit with the benchhub-client.

Metric

A metric is a Python function that compares a prediction to ground truth and returns a number. Metrics live in a reusable library (a global metric) and are bound to a leaderboard with:

  • argument mappings — which dataset/prediction field feeds each function argument;
  • pooling — how per-sample scores aggregate into the leaderboard number (mean, median, percentile, min, max);
  • sort direction — higher- or lower-is-better;
  • an optional tag filter to score only a slice of samples.

A metric's signature can request typed inputs (def accuracy(gt: bh.Label, pred: bh.Label) — bh is the benchhub package (import benchhub as bh), and bh.Label, bh.Depth, etc. are the typed kinds). Metric outputs can chain — one metric can consume another's per-sample output. Re-scoring existing submissions needs no re-upload (a recompute button on the leaderboard). See Writing Metrics.

Why a reusable library + per-leaderboard binding: one well-written metric (say IoU) is authored once and bound to many leaderboards, each mapping it onto its own fields — you don't rewrite scoring per benchmark.

Visualization

A visualization is a Python function that renders an image for a sample (per-sample) or for a whole submission (aggregated, e.g. a confusion matrix). Like metrics, they live in a library and bind to a leaderboard. See Visualizations.

The comparison view

This is the payoff. The comparison view lines up ground truth against each submission's prediction for every sample, side by side — images, masks, depth colormaps, video sequences, confusion matrices, per-sample metric charts. Pick any subset of submissions and read them column by column, sample by sample.

Why it matters: a leaderboard alone is a single number per model. The comparison view turns that number back into an inspection tool — you see exactly where a model wins or loses, not just its aggregate score. This is the insight that one-off benchmark scripts throw away, and the reason the typed contract is worth declaring: because every field's kind is known, the platform can render ground truth and prediction the same way and stack them up automatically.

The sandbox

All user-supplied code — metrics, visualizations, and custom-type rendering — runs in a hardened Docker sandbox container, with typed arguments and any bytes crossing the boundary as JSON.

Why: scoring and rendering should be open to everyone, but no one's code should be trusted. The sandbox lets any user contribute a metric or visualization that runs against other people's data without the platform having to vet it first.

Visibility

Datasets, leaderboards, metrics, and visualizations each have a visibility:

  • public — appears in the catalog, visible to everyone;
  • unlisted — reachable by link, hidden from catalogs;
  • private — only you (and admins) can see it.

Your own imports and submissions default to private; you publish when ready. Publishing pre-checks your public storage budget.

Quotas

Storage is split into two budgets so working space and shared content don't compete:

  • Public — 50 GB by default, charged for anything whose visibility is public.
  • Private — 10 GB by default, for private/unlisted working content.

Because datasets are cached as the small preview tier, imports are cheap; the real cost is per-leaderboard materialization. Submission ZIPs aren't charged to you — the leaderboard owner already paid for the inputs you respond to.

A note on “projects.” Older URLs are prefixed with a project name, but datasets and leaderboards are global today — you can ignore the project concept.
