Core concepts

The handful of objects everything else is built from.

The one big idea: a typed contract. Every piece of data — a dataset column, a leaderboard input, a prediction — has a declared kind (image, depth, mask, label, sequence, …). Because both the dataset and the submitter agree on kinds, the platform can decode, render, and score data without guessing. The kinds are documented in Data Types.

Dataset

A dataset is a global collection of samples (it is not scoped to any project or user namespace — any leaderboard can bind any dataset). Each dataset has a set of fields, and each sample holds one value per field.

Datasets are stored in a two-tier way:

  • Preview tier (always present): a small, downscaled rendering of every sample — JPG/PNG thumbnails for images/masks/depth, a waveform for audio, the inline value for text/labels/scalars. This is what you see when you browse a dataset. It is tiny (tens of KB per sample), so an imported dataset barely uses storage.
  • Materialized tier (per leaderboard, on demand): the full-resolution bytes for the subset of samples a leaderboard actually uses. See Materialization below.

Fields & roles

A field has a name, a kind, and a role. The role says how the field participates in a benchmark:

RoleMeaning
inputGiven to the submitter at inference time (e.g. the RGB image). Not predicted.
gtGround truth, held server-side — the target a prediction is scored against.
predA field the submitter produces. Declared on the leaderboard's prediction contract.

Roles are decided per leaderboard, so the same dataset can be used different ways. A handy default: if a dataset has an image field and nothing else is marked input, the image is treated as the input and the rest become ground truth.

Leaderboard

A leaderboard binds one (or more) datasets and defines a benchmark:

  • Which fields are inputs vs ground truth (the field roles above).
  • A prediction contract — the list of fields a submitter must produce, each with a name (<x>_pred) and a kind. The contract is frozen once verified submissions exist, so existing predictions are never re-interpreted through a different decoder.
  • Metrics and visualizations bound to it (see below).
  • A materialized sample subset (see below).

The leaderboard page shows a ranked table of submissions; the comparison view opens any subset of them side-by-side, per sample.

Materialization

Browsing only needs the cheap preview tier, but scoring needs the real bytes. So when you create a leaderboard you choose a sample subset to materialize at full resolution:

  • Sample cap — how many samples (or all of them).
  • Sampling — head (first N), random, or stratified (balanced across a label field). Random is the default; stratified is auto-suggested when the dataset has a label field.
  • Seed — makes random/stratified picks reproducible.

Materialization runs in the background; the leaderboard page shows its status (pending → running → ready, or failed with a retry). The materialized bytes count against the leaderboard owner's storage quota. Re-materializing is blocked once the leaderboard has submissions — changing the sample set would invalidate the inputs those submissions were scored against.

Submission

A submission is one model's set of predictions for a leaderboard's samples. Submissions are verified (uploaded through the real predict + eval pipeline) — the only kind you create. Each carries a processing status (Pending → Processing → Processed, or Error) and, once scored, a row of metric results. You submit with the benchhub-client.

Metric

A metric is a Python function that compares a prediction to ground truth and returns a number. Metrics live in a reusable library (a global metric) and are bound to a leaderboard with:

  • argument mappings — which dataset/prediction field feeds each function argument;
  • pooling — how per-sample scores aggregate into the leaderboard number (mean, median, percentile, min, max);
  • sort direction — higher- or lower-is-better;
  • an optional tag filter to score only a slice of samples.

A metric's signature can request typed inputs (def accuracy(gt: bh.Label, pred: bh.Label)) and metric outputs can chain — one metric can consume another's per-sample output. See Writing Metrics.

Visualization

A visualization is a Python function that renders an image for a sample (per-sample) or for a whole submission (aggregated, e.g. a confusion matrix). Like metrics, they live in a library and bind to a leaderboard. See Visualizations.

Visibility

Datasets, leaderboards, metrics, and visualizations each have a visibility:

  • public — appears in the catalog, visible to everyone;
  • unlisted — reachable by link, hidden from catalogs;
  • private — only you (and admins) can see it.

Your own imports and submissions default to private; you publish when ready. Publishing pre-checks your public storage budget.

Quotas

Storage is split into two budgets so working space and shared content don't compete:

  • Public — 50 GB by default, charged for anything whose visibility is public.
  • Private — 10 GB by default, for private/unlisted working content.

Because datasets are cached as the small preview tier, imports are cheap; the real cost is per-leaderboard materialization. Submission ZIPs aren't charged to you — the leaderboard owner already paid for the inputs you respond to.

A note on “projects.” Older URLs are prefixed with a project name, but datasets and leaderboards are global today — you can ignore the project concept.
