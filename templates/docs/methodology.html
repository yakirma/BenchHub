How we score

Why a number on BenchHub means something — and how it stays apples-to-apples across every model on a board.

The problem with self-reported scores

Most leaderboard numbers come from papers and model cards — each produced with a different eval set, prompt, decoding setup, and scoring script. They rarely reproduce and barely compare. BenchHub fixes the comparison by making every model on a board run the same protocol, scored by us.

1. Same data, same metric

Every board binds one eval set and one metric. Each submission is scored on the identical samples with the identical scorer — so a difference in the score is a difference in the model, not in the harness. The metric is declared once (typed, in Python) and applies to every submission equally.

2. Sandboxed, not self-reported

You submit predictions, not a number. BenchHub decodes them against the board's typed contract and runs the scoring metric server-side in a hardened container — no network, no access to held-out ground truth beyond the scorer. The score is computed here, on your actual outputs, so it's reproducible by construction rather than taken on trust.

3. LLM boards pin the exact protocol

For LLM benchmarks (MMLU, HellaSwag, ARC-Challenge, GSM8K) the eval protocol is baked into the input: the exact prompt — question, lettered choices, and answer instruction — is part of the dataset, so every submitter sends the identical string. Answers are scored by exact match (e.g. the chosen A–D letter), zero-shot, with no per-model prompt tuning. That removes the biggest source of "leaderboard inflation": prompt and few-shot cherry-picking.

4. A visible trust gradient

Boards distinguish verified submissions (scored by BenchHub from predictions) from any reported figures mirrored from external sources. Verified beats reported — and only verified scores feed the cross-board model rankings, which combine a model's normalized score across every board in a sub-category.

5. Open and re-runnable

Standings are public and mirrored read-only to a Hugging Face Space. The eval sets are real datasets, the metrics are inspectable Python, and anyone can submit the same way — so a result is something you can reproduce, not just read.
