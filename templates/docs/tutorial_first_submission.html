Your first submission

The canonical loop, walked end to end once: a tabular HuggingFace dataset (image + label) → a leaderboard → a first scored submission — on CIFAR-10, scored by accuracy.

Every benchmark on BenchHub is the same four moves: import a dataset, create a leaderboard that declares what to predict and how to score it, submit predictions from Python, then compare them against ground truth. The data source changes from benchmark to benchmark; the shape doesn't. Learn it here on the simplest possible case — classify 32×32 images into ten classes — and every other tutorial is a variation on it.

New to the vocabulary (kind, field, ground truth, contract)? Core Concepts builds the object model bottom-up. This page defines each term as it comes up, so you can also just read straight through.

Before you start: you need an account — sign in (top-right) first. Importing a dataset, creating a leaderboard, and getting an API token all require it.

1 · Import the dataset

An import pulls a public dataset onto BenchHub as a browsable, typed Dataset. CIFAR-10 lives on HuggingFace as a clean table — one row per image, with an img column and a label column — which is the simplest layout to import: no file-tree mapping, just confirm the inferred columns.

  1. Open the importer. Go to Datasets → Add a dataset, paste the HuggingFace repo id uoft-cs/cifar10, and click Fetch & preview. BenchHub reads the dataset's schema and shows you what it found before anything is committed.
  2. Confirm the kinds. Each column is assigned a kind — its declared type, the single decision the whole system keys off. You'll see the image column inferred as image and the label column as label (an integer drawn from a fixed class vocabulary, rendered as a class name in the UI rather than a bare index). If a kind looks wrong here, fix it now — decode, render, score, and compare all fall out of it downstream.
  3. Pick the split. Choose the test split. You score against held-out data, never the data a model trained on. Leave max samples at your taste — the full split is fine; you'll narrow it down per-leaderboard in the next step, not here.
  4. Commit. The dataset caches as a preview: a tiny, thumbnail-resolution copy (downscaled images, inline labels) meant for fast browsing, not scoring. It's a few tens of KB per sample, so the import is quick and cheap. Full- resolution bytes don't land yet — they only arrive for the exact samples a leaderboard actually scores on (see step 2). New datasets default to private; you publish when you're ready.

Not a tabular repo? Paired files, folder-as-label, video clips, and other non-table layouts go through the file-tree mapper instead — see Data layouts. Already have data in memory? Build a dataset in Python. Either way you land back on this same loop.

2 · Create the leaderboard

A dataset is just data. A leaderboard turns it into a benchmark: it picks which samples to score on, declares the prediction contract (what submitters must hand back), and binds a metric (how predictions are scored). Open the dataset and click New leaderboard.

Field roles — input vs. GT

Every dataset field carries a role that says how it's used at scoring time:

  • input — given to the submitter as conditioning, and not predicted. Here, image = input: the model sees each image and must produce an answer for it.
  • gt — the ground truth (GT): the held-aside correct answer, kept server-side and revealed only in the comparison view. Here, label = gt: it's what each prediction is scored against, so submitters never receive it through the client.

This split is what keeps the benchmark honest — the client hands out inputs and withholds GT automatically, purely from the roles you set.

The prediction contract

The contract is the list of fields a submission must produce, each with a name and a kind. For classification it's a single field: label_pred, kind label. The kind is load-bearing — declaring label_pred as label tells the client to accept a bh.Label(...) and tells the metric engine to decode it as a class index. The _pred suffix is just the convention pairing it with the label GT field it'll be scored against. You can inspect any board's contract from code with lb.contract() — a list of {name, kind, params, role} entries.

Sample selection — stratified sampling

You rarely need the whole test split to rank models. Pick 1,000 samples, stratified by label. Stratified sampling draws proportionally from each class, so all ten CIFAR classes are represented evenly instead of whatever the head of the file happens to contain. (Plain head/random sampling can skew when a split is sorted by class.) A smaller, balanced subset scores fast and still ranks models faithfully.

Bind the metric

A metric is the scoring function. Bind the built-in accuracy — it ships with BenchHub, so there's no code to write — and map its arguments to your fields: gt→label, pred→label_pred. That mapping is how the engine knows which GT field and which prediction field to feed the metric for each sample.

Create — and what materialization does

Click Create. Now materialization kicks off in the background: BenchHub downloads the full-resolution bytes for exactly the 1,000 samples this leaderboard scores on (the preview tier from step 1 is too low-res to score or to serve to a model). It runs as a background job because fetching full-res media for a thousand samples takes a moment, and there's no reason to block the page on it — the board exists immediately; the status flips to ready once the bytes land. You only pay the storage cost for the subset a board actually uses, not the whole dataset.

Give materialization a moment to finish before submitting. iter_samples serves a board's materialized subset; if it's still downloading, wait for the board to show ready.

3 · Submit predictions

Submissions come from the benchhub-client Python package. Install it with pip install benchhub-client, then grab an API token from Settings → API tokens → Generate (or your account page → API tokens). Tokens are copy-once — copy it when shown; Regenerate immediately invalidates the old one.

Use your board's handle, shown on its page as “Client handle” — alice/cifar10-1k below is just an example. The loop:

import benchhub as bh
client = bh.Client(token="<your-api-token>")
# The board's handle, shown on its page as "Client handle" (an id still works too):
lb = client.leaderboard("alice/cifar10-1k")

sub = lb.submission(name="my-classifier")
for name, inputs in lb.iter_samples():
    x = inputs["image"].array        # (32,32,3) uint8
    sub.predict(name, label_pred=bh.Label(int(my_model(x))))
sub.submit(description="baseline CNN")

Reading it line by line:

  • bh.Client(token=…) — you can also drop the argument and let the client read $BENCHHUB_API_TOKEN from the environment. The base URL defaults to https://runbenchhub.com (override with $BENCHHUB_BASE_URL).
  • client.leaderboard("alice/cifar10-1k") — references the board by its handle (<owner>/<slug>), shown on the board page as “Client handle”. An integer id works anywhere a handle does.
  • lb.iter_samples() — yields (name, inputs) for every sample in the materialized subset. inputs is keyed by input-field name, and each value arrives already decoded: the image field is a bh.Image, so inputs["image"].array is a (32,32,3) uint8 NumPy array. File-backed inputs download once as one bulk archive cached under ~/.cache/benchhub/ (override with $BENCHHUB_CACHE_DIR); re-runs read from disk with no network, and force_download=True forces a re-fetch.
  • sub.predict(name, label_pred=bh.Label(…)) — stages one typed prediction per sample. The keyword label_pred and the bh.Label wrapper must match the contract from step 2; the client validates immediately.
  • sub.submit(…) — packs the staged predictions and uploads them. description (and optional link) show on the submission row. Guard against re-uploading the same run with lb.submission_exists(name).

Replace my_model with your classifier. See Submitting for the full client reference.

4 · Compare

After submit() the server validates your predictions against the contract and queues scoring; the run moves Pending → Processing → Processed. Your submission then appears on the leaderboard with its accuracy and rank. Click it — or hit the comparison view — to line up ground truth against your prediction, sample by sample.

Why the comparison view matters. The leaderboard gives you one number; the comparison view lines up the GT against each model's prediction, sample by sample. So you don't just learn that a model scored 0.84 — you see exactly which samples it got wrong, and how. That's where the score turns into insight about where your model fails.

Changed the metric after submitting? You don't re-upload — the board owner hits Recalculate (↻) from the Actions menu to re-score every submission, or the per-row ↻ button to re-score just one.

This is the whole loop. Bring data in (the import), declare the prediction contract (label_pred, kind label), bind a metric (accuracy), submit & compare. Every other tutorial swaps out one piece — a different data layout, a custom metric, a different kind — without changing the shape.
