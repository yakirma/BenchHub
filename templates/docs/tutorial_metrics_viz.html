Author a metric & visualization

A leaderboard's score and its comparison view (the side-by-side per-sample GT-vs-prediction table) are both just Python you write, bind, and re-run. This tutorial builds an accuracy metric from scratch, binds a ready-made confusion-matrix view, and re-scores existing submissions without anyone re-uploading.

Earlier tutorials brought data in and pushed predictions. This one fills in the missing half: how the number gets computed. The dataset and submissions stay put — you change scoring on the server side and apply it retroactively. Nothing here requires the submitters to lift a finger.

The mental model. A metric is a function that scores one prediction against ground truth (GT). A visualization is a function that draws a picture from the same fields. You author each in the library, then bind it to a leaderboard — the binding is what wires the function's arguments to actual dataset/prediction fields. The same metric can be bound to many boards with different mappings.

1 · Write a metric on the Metrics page

Open the Metrics page and create a new metric. A metric is plain Python source: it takes one argument per mapped field and returns a number. numpy (as np) and benchhub (as bh) are pre-injected; import anything else inside the body. (accuracy already ships as a built-in metric — here you rebuild it from scratch to learn authoring.) Paste this:

import benchhub as bh

def accuracy(gt: bh.Label, pred: bh.Label):
    return 1.0 if pred.value == gt.value else 0.0

Two things in that signature are doing real work — this is the part to understand:

  • The type annotations declare accepted kinds. gt: bh.Label and pred: bh.Label are parsed at save time into the metric's accepted kinds (a kind is a field's declared type — label, image, depth, …). Because the args are typed, BenchHub hands you a decoded bh.Label instance (read .value for the class index) instead of a raw primitive. Annotate with a | union (pred: bh.Label | bh.LabelList) to accept more than one kind; an un-annotated metric keeps receiving the raw value for back-compat.
  • The argument names map to roles by convention. Name an argument gt, pred, or input and BenchHub knows which role it expects — gt is the held-aside answer, pred the submitted prediction, input a conditioning field that isn't scored. This is why the binding step below knows that gt should map to a GT-role field and pred to a prediction field.
Per-sample vs aggregated. The metric above returns one number for one sample — BenchHub runs it on every sample and pools the results into the single leaderboard number. Each per-sample value is also stored, so re-pooling is instant and never re-runs your code. (Metrics that can't be averaged sample-by-sample — AUROC, corpus BLEU — instead receive the full lists of values and return one number; see Writing metrics.)

Bind it to a leaderboard

Saving the metric only adds it to the library — it scores nothing until you bind it to a leaderboard on that board's Edit page. Binding takes the abstract function and pins each argument to a concrete field:

Authoring a metric or visualization is open to everyone; binding one to a board and re-scoring (Recalculate) are owner/admin-only.

  1. Argument mappings. Map gt→label and pred→label_pred — i.e. the function's gt arg reads the dataset's label field, and pred reads the submission's label_pred prediction field. (The argument names from step 1 are why the form pre-suggests a GT field for gt and a pred field for pred.) The arg's declared kind must match the field's kind, or there's no valid field to point it at.
  2. Display name. The column header on the board — and the handle other metrics can chain on (see below).
  3. Pooling. How the per-sample scores pool into the single board number: mean, median, percentile, min, or max. mean is the common choice — accuracy is a mean of 0/1 values.
  4. Sort direction. Higher-is-better (accuracy) or lower-is-better (an error metric) — this decides which way the leaderboard ranks.

Optionally add a tag filter to score only a subset of samples. Once you save the binding, every existing submission's stored per-sample predictions feed the metric and the column appears.

2 · Add a confusion-matrix visualization

The leaderboard gives you one number; a visualization turns the same fields into a picture. A confusion_matrix aggregated viz ships ready to bind — no code to write:

  1. On the leaderboard's Edit page, bind confusion_matrix and map gt→label, pred→label_pred — the same two fields the metric uses. Its axes pick up the class names automatically.
  2. Aggregated means it draws one figure for the whole submission (every GT label vs every predicted label), not one per sample — exactly what a confusion matrix needs.

Prefer your own? Author a visualization the same way you authored the metric — a typed Python function that returns a figure — on the Visualizations page. The argument-kind and role conventions are identical to metrics. See Visualizations for the figure API and the per-sample vs aggregated split.

3 · Re-score existing submissions

You changed scoring after submissions already landed — and that's the whole point. No one re-uploads anything.

On the leaderboard, re-scoring lives behind the Recalculate ( ↻) control: the per-row ↻ button re-scores a single submission, and Actions → Recalculate re-scores every submission on the board. No re-upload needed. BenchHub replays each submission's stored per-sample predictions through the new/changed metric bindings, re-pools, and refreshes the columns and bound visualizations. Because predictions are stored decoded, this never re-runs the submitters' models — it only re-runs your scoring code.

Use this any time you tweak a metric, change pooling, fix a bug in scoring, or bind a brand-new metric to a board that already has runs.

Two things worth knowing

Metric outputs can chain

A metric can consume another metric's per-sample output by mapping one of its arguments to that metric's display name instead of a dataset field. BenchHub topologically sorts the bound metrics so dependencies run first, then feeds the upstream values in — so you can build, say, a normalized or thresholded score on top of a raw error without recomputing the error.

All user code runs in a hardened sandbox

Both metric and visualization code execute inside a network-isolated, read-only container (one short-lived sandbox per scoring job) — never in-process on the server. That's why anyone can author and publish scoring code: it can't reach the network, the DB, or other submissions' raw bytes. Decoded typed arguments and any image bytes cross into the sandbox as data, and results cross back out.

A metric is just code — spot-check anything non-trivial (rank-based, span-overlap, BLEU-family) on a couple of samples in the comparison view before you trust the column. The sandbox makes user code safe to run, not automatically correct.
