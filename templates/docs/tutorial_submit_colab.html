Submit from Google Colab

Run the BenchHub submission loop in a Google Colab notebook on a free GPU — no local Python, no CUDA install. The board hands you a pre-filled notebook; you fill in one function and hit Run all.

Why Colab

The submission loop (install the client → resolve a token → iterate samples, run your model, predict, submit) is identical everywhere. Colab is the lowest-friction place to run it:

  • Zero install. No local Python, no CUDA, no driver mismatch — the runtime ships ready, and the notebook pip installs benchhub-client in its first cell.
  • Free GPU. Flip Runtime → Change runtime type to a GPU and your model runs on hardware you didn't pay for.
  • The board does the boilerplate. BenchHub generates a notebook tailored to one leaderboard — it already names the board's handle and wires up the right bh.<Kind>(…) constructors for that board's prediction fields. You only write the model.
New to the loop itself? Submitting predictions walks the iterate → predict → submit flow and the typed bh.<Kind> instances. This page is purely about running it in Colab.

1 · Get the notebook

Open the leaderboard you want to submit to. Its page has a Submit block with three tailored actions — all three are generated for this specific board:

  • Open in Colab (green) — one click: the server publishes the tailored notebook to a GitHub gist and opens it in Colab, pre-filled and ready. Path A below.
  • Download notebook (.ipynb) — the same tailored notebook as a file, to upload into Colab by hand. Path B below (and the fallback when one-click isn't available).
  • Download script (.py) — a self-contained Python script that reads BENCHHUB_API_TOKEN from the environment. For running locally or on your own box — see Submit from a script.

2 · Path A — one-click “Open in Colab”

This is the fast path. The button needs your identity to mint a token, so sign in to BenchHub first — or just click Open in Colab while signed out: it sends you through login and returns you to the same action automatically. Then:

  1. The server publishes the board-tailored notebook to a GitHub gist and redirects you straight into Google Colab with it loaded.
  2. Baked into the notebook is a short-lived submission token — it expires in about 1 hour. That's long enough to run a submission right now without touching Settings.
  3. Sign in to Colab with any Google account, fill in your model, and run. No token wrangling needed for this session.
The one-click path needs a GitHub gist token configured on the server. If it isn't, the board flashes a note instead of opening Colab: download the .ipynb and upload it by hand. That's Path B — identical notebook, one extra step.

3 · Path B — manual upload (fallback)

Use this when one-click isn't available, or any time you'd rather not go through a gist:

  1. Click Download notebook (.ipynb) in the Submit block.
  2. Go to colab.research.google.com and choose File → Upload notebook, then pick the .ipynb you just downloaded.

It's the same tailored notebook as Path A — the only difference is that a manually-uploaded notebook has no inline token baked in, so token resolution falls through to the Colab Secret or the interactive prompt (see the next section).

4 · The three cells

After a title cell, the generated notebook has exactly three code cells. Run them top to bottom.

Cell 1 — install

!pip install -q --upgrade benchhub-client numpy

Pulls the client into the Colab runtime. Quiet (-q) and always upgrades so you get the current client.

Cell 2 — token resolution

This cell finds an API token and sets os.environ['BENCHHUB_API_TOKEN'] so the client picks it up automatically. It tries three tiers in order, and validates the winner against /api/whoami before continuing, so a stale or wrong token fails loudly here rather than at submit time:

  1. Inline token — the short-lived (≈1 hour) token baked in by Open in Colab. Present only on a Path A notebook; absent after a manual upload.
  2. Colab Secret named BENCHHUB_API_TOKEN — a durable token you set once (next section). This is what makes the notebook reusable past the 1-hour window.
  3. Interactive prompt — if neither of the above resolves, a getpass box asks you to paste a token. Nothing is echoed to the notebook output.
Inline token vs. durable Secret. The inline token is convenient but ephemeral — once it expires (~1h) re-running the notebook fails token validation. The Colab Secret is the durable answer: set it once and every future run resolves to it, no matter how old the notebook is.

Cell 3 — the body

The board-tailored script. It references the leaderboard by its handle, builds a submission with sub = lb.submission(), gives you a my_model(inputs) stub to fill in, and runs the iterate → predict → submit loop with the bh.<Kind>(…) constructors that match this board's prediction fields. The snippet below is illustrative of that generated body — the real cell already has the handle and the matching constructors wired in for your board:

import benchhub as bh

client = bh.Client()                            # reads BENCHHUB_API_TOKEN from the env (cell 2)
lb = client.leaderboard("<owner>/<slug>")    # the board's handle, pre-filled for you

# print(lb.contract())   # [{name, kind, params, role}, ...] — what this board expects

def my_model(inputs):
    # TODO: your model. inputs[<field>] are decoded bh.<Kind> instances:
    #   image/mask/depth -> bh.Image/bh.Mask/bh.Depth (use .array)
    #   audio -> bh.Audio ; sequence -> iterable clip
    #   scalar/label/text/json/label_list -> the raw decoded value
    ...

sub = lb.submission(name="my-colab-run", description="...", link="...")
for name, inputs in lb.iter_samples():
    pred = my_model(inputs)
    sub.predict(name, label_pred=bh.Label(int(pred)))   # constructor(s) tailored to this board
sub.submit()

You must return a value for every prediction field in lb.contract() on every sample — a sample missing any required pred field fails validation.

5 · Reuse it — add a Colab Secret

The inline token dies in an hour. To keep re-running this notebook (or any BenchHub notebook) for days, give Colab a durable token once:

  1. In Settings → Account → API tokens (also at /settings/api_tokens), click Generate token (or Regenerate if you already have one). Copy the token — it has no special prefix, and it's shown only once. Regenerate immediately invalidates the old token, so anything still using it — other notebooks, scripts — stops working until you update it.
  2. In Colab, open the key icon ( Secrets) in the left sidebar.
  3. Add a secret named exactly BENCHHUB_API_TOKEN with the token as its value.
  4. Toggle Notebook access on for it — otherwise the notebook can't read it and tier 2 falls through to the prompt.

Now cell 2 resolves to the Secret on every run, regardless of whether the inline token has expired. Set the Secret once and forget about tokens.

6 · Fill in the model, pick a GPU, run

  1. Write my_model() in the body cell. Inputs arrive as already-decoded bh.<Kind> instances — inputs["image"].array is a NumPy array, an audio input is a bh.Audio, a sequence input is an iterable clip, and inline kinds (scalar/label/text/json/label_list) are the raw decoded value. Return predictions wrapped in the bh.<Kind> constructors the body cell already shows for this board.
  2. Pick a GPU. Runtime → Change runtime type → choose a GPU accelerator, then save.
  3. Run all. Runtime → Run all. Cells 1–2 set up the client and token; cell 3 iterates the board's samples and submits.
File-backed inputs (images, masks, depth, audio, clips) download once as a single bulk zip, cached under ~/.cache/benchhub (override with $BENCHHUB_CACHE_DIR). On a Colab runtime that cache lives only as long as the session — a fresh runtime re-downloads. Pass force_download=True to iter_samples to force a re-fetch.

7 · After it submits

  • The run appears on the leaderboard with its score and rank; status moves Pending → Processing → Processed as metrics finish.
  • Open the comparison view to inspect GT vs. your prediction sample-by-sample — that's where a single score turns into insight about which samples your model got wrong.
  • Iterating? Edit my_model() and re-run just the body cell (cells 1–2 already ran) to resubmit — no need to Run all again.
  • Give each run a distinct name. Guard against duplicates with lb.submission_exists(name): 
    name = "my-colab-run"
if lb.submission_exists(name):
    name = name + "-v2"
sub = lb.submission(name=name)

Troubleshooting

  • Token rejected (/api/whoami fails in cell 2). The inline token has likely expired (~1h). Generate a fresh one at Settings → Account and either paste it at the prompt or, better, set it as the BENCHHUB_API_TOKEN Colab Secret so it sticks.
  • “Open in Colab” landed on Colab's homepage (no notebook). The server has no gist token configured. Use the .ipynb upload path — Download notebook (.ipynb), then File → Upload notebook in Colab.
  • GPU out of memory. Use a smaller batch size in my_model(), or process one sample at a time. The loop already hands you samples one by one — batch only as much as the GPU holds.
