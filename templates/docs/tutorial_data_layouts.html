Bring your own data layout

Real datasets rarely arrive as a clean table — they're trees of paired files, packed arrays, folders that are the label, or clips of frames. These four recipes map each of those shapes onto BenchHub's typed contract so the rest of the loop — submit, score, compare — works unchanged.

The file-tree mapper

BenchHub's importer infers columns automatically for tabular HuggingFace repos. But plenty of datasets aren't tabular: they're a directory tree where structure lives in the paths — paired files (an RGB next to its depth map), arrays packed into one .npz, a folder name that encodes the class, frames grouped into clips. For those, the file-tree mapper takes over.

You don't pick it manually. On a non-tabular repo, Fetch & preview detects there's no flat schema and hands off to the file-tree mapper automatically. The mapper lists the tree, surfaces the distinct path levels, and lets you declare one field per row — each row says how to pull that field out of the tree:

Per-field anatomy — every mapper row carries:
  • loader — how the bytes are read. One of:
    • file — one file per sample (an image, an audio clip).
    • npz — an array inside a .npz archive, addressed by key; can be shared across samples and sliced along an axis.
    • token — a value lifted straight out of the path (a folder name becoming a label), no file read at all.
    • sequence — many files gathered into one ordered clip.
  • pattern — the path template, with {tokens} naming the parts that vary. The token that identifies the sample ties a field's files to the same row across the dataset.
  • kind — the field's declared type (image, depth, label, sequence, …); see Data types.
  • role — the field role: input (conditioning handed to the submitter, never predicted) or gt (the held-aside answer a prediction is scored against). Get this right: a field marked input won't appear as a prediction target. (Don't confuse this with the mapper's path-level types — id / property / token — which describe how a path segment is read, not what a field is for.)

The index rule: the first file field enumerates the samples — each distinct {id} it matches is one sample, and every other field is pulled in alignment to that row.

After you map the fields and create the board, the submission loop is identical to every other tutorial — iterate samples, run your model, predict, submit. Two of the recipes below show the loop because the prediction kind is interesting (depth, sequence); for the full reusable loop and the Colab path, see Submit from a script and Submit from Colab.

Imported datasets cache as the lightweight preview tier (thumbnail-resolution, for browsing) and default to private. Full-resolution bytes only land when a leaderboard materializes the subset it scores on. You publish when ready.

Paired RGB + depth (file tree)

A repo laid out as train/<seq>/normal/<frame>.png for the RGB frames (the normal/ level is the quality variant), plus one packed depth.npz per sequence holding every frame's depth map. This is the canonical paired-files-plus-packed-array shape, and it's not tabular — the file-tree mapper handles it.

  1. Inspect. Paste the repo id and click Fetch & preview. Because there's no flat schema, it hands off to the file-tree mapper, which lists the tree and the path levels it found.
  2. Map fields. 
    image : loader=file  pattern=train/{seq}/normal/{id}.png   role=input  kind=image
depth : loader=npz   pattern=train/{seq}/normal/depth.npz  key=depth  shared=yes  axis=0
        role=gt  kind=depth
    The RGB row reads one .png per sample. The depth row points every sample at the same archive (shared=yes): all frames are stacked along axis=0 under the depth key, and BenchHub slices it so frame N aligns to the Nth image in sorted filename order. This is why packed arrays don't need one file per sample — one .npz fans out across the sequence.
  3. Decode preview. Click a sample and confirm the depth frame actually lines up with image 0 before committing — a wrong axis or sort order is the one thing that silently misaligns paired data.
  4. Leaderboard + metric. Create a leaderboard with image=input, depth=gt, a prediction field depth_pred (kind depth), and bind rmse_depth mapped gt→depth, pred→depth_pred.
  5. Submit. Depth is role=gt, held server-side — you never see it. The only input is the image, which arrives as a decoded bh.Image instance; read its pixels with .array and predict the depth: 
    import benchhub as bh
client = bh.Client()   # reads $BENCHHUB_API_TOKEN

lb = client.leaderboard("alice/rgbd-depth")   # the board's handle
sub = lb.submission(name="my-depth-model")
for name, inputs in lb.iter_samples():
    rgb = inputs["image"].array
    sub.predict(name, depth_pred=bh.Depth(my_depth_model(rgb), unit="meters"))
sub.submit()

Folder-as-label classification

A repo where the folder name is the class — Alex_Brush/000.png, Cookie/001.png, … The label never appears in a file; it lives in the path. The token loader exists for exactly this.

  1. Describe the structure. In the mapper, the “describe the structure” panel shows two path levels, each with a path-level type (distinct from a field's input/gt role): the class folder (default property) and the filename (id). Click Generate fields from structure and the mapper writes both rows for you.
  2. It produces: 
    image : loader=file   pattern={label}/{id}.png   kind=image  role=input
label : loader=token  token=label                kind=label  role=gt
    The image row reads each file as before. The token loader reads no bytes at all — it turns each folder name into a label and builds the class vocabulary across the whole dataset, so the UI shows Cookie rather than a bare index.
  3. Leaderboard. The image is role=input and the label is role=gt. Add a prediction field label_pred (kind label); bind accuracy mapped gt→label, pred→label_pred.
  4. Submit. The loop is the first-submission loop verbatim — iterate, classify inputs["image"].array, predict label_pred=bh.Label(…), submit.
Folder-as-label is the most common “ImageFolder” layout in the wild. The token loader generalizes past labels: any path segment you want surfaced as a typed field — a split name, a category, a source — can ride the same row.

Video / clip (sequence) benchmark

Each sample is a clip — many frames that belong together — e.g. clips/<id>/<frame>.png. The sequence loader gathers the frames into one ordered unit per sample.

  1. Map a sequence field. 
    clip : loader=sequence  pattern=clips/{id}/{frame}.png   kind=sequence  role=input
    The {frame} token marks the per-frame part; the {id} token identifies the sample. So each distinct {id} becomes one sample, and all its frames are gathered into a clip in sorted frame order. In the dataset and comparison views the clip renders as an mp4.
  2. Predicting a sequence. A sequence isn't only an input kind — it can be a prediction target too. Give the leaderboard a prediction field clip_pred (kind sequence). On the client, each input clip is iterable: loop its frames, run your per-frame model, and wrap the results back into a bh.Sequence: 
    lb = client.leaderboard("you/your-clip-board")   # the board's handle
sub = lb.submission(name="my-frame-model")
for name, inputs in lb.iter_samples():
    clip = inputs["clip"]                                    # clip is iterable
    out = [bh.Image(my_frame_model(f.array)) for f in clip]
    sub.predict(name, clip_pred=bh.Sequence(out, item_kind="image", fps=10))
sub.submit()
    item_kind declares what each element is (here image); fps sets the playback rate for the rendered mp4.

Variants & splits

Sometimes a file-tree repo encodes a quality level or a split inside the path — e.g. train/<seq>/{quality}/<id>.png, where {quality} is low or normal. The mapper gives you two ways to handle that extra path token, depending on whether you want one dataset or many.

  • Import just one split/subset — set the mapper's “import only where token = value” to e.g. quality = normal. You get one dataset containing only those samples; the other variants are skipped.
  • Fan out into separate datasets — set the variant token to quality. BenchHub creates one dataset per distinct value (<name>_low, <name>_normal), each importing only its own slice. Useful when you want a separate leaderboard per condition.

Either way the path token never leaks into the field contract — it's a selection lever, not a field. The samples you keep still map through the same loader/kind/role rows as the recipes above.

Stuck on a specific repo's layout? Open a GitHub issue with the repo id and structure — odd trees are how the mapper grows.
