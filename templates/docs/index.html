BenchHub documentation

ML benchmarking is a pile of one-off glue — every benchmark reinvents its data layout, scoring, and submission format, and hands you back a single number with no insight into where your model fails. BenchHub's fix is one idea: declare the data contract once, and make it typed.

What is BenchHub?

BenchHub is an open benchmarking platform — a web app plus a Python package (benchhub-client, imported as import benchhub as bh). You bring a dataset, turn it into a leaderboard with a clear typed prediction contract, and let anyone submit a model's predictions to be scored by metrics you choose. Every submission is ranked, and any of them can be opened in a deep comparison view that lines up ground truth against each prediction, sample by sample.

That comparison view is the payoff. Images, masks, depth colormaps, video sequences, confusion matrices, and per-sample metric charts all sit side by side — ground truth (the correct answer held server-side that a prediction is scored against) next to each prediction — so a leaderboard stops being a single number and becomes an inspection tool. You see where a model wins or loses, not just its aggregate score.

The one big idea: a typed contract

BenchHub makes the data contract explicit and typed, once.

Every piece of data — a dataset field, a leaderboard input, a prediction — carries a declared kind: image, mask, depth, audio, text, bboxes, label, scalar, json, sequence (an ordered clip of frames that renders as an mp4), and more — and you can register your own. Because the dataset, the client, and the metric engine all agree on the kind, the platform never has to guess: decode, render, colormap, score, and compare all fall out of that single decision. If both sides agree a field is a depth map in meters, it is handled the same way every time — across datasets, models, and metrics.

Everything downstream is a consequence. Two-tier storage keeps a cheap preview of every sample so you browse a whole dataset without paying for full-resolution bytes. Per-leaderboard materialization pulls full-res bytes only for the subset a benchmark scores on, charged to its owner, so cost stays proportional to use. A hardened Docker sandbox runs all user-supplied metric and visualization code, so anyone can contribute scoring without being trusted. The typed contract is the spine; the rest follows. The authoritative kind list lives on the Data Types page.

The end-to-end flow

A typical lifecycle on BenchHub has four stages:

1
Import a dataset

Pull a HuggingFace repo (tabular or file-tree), or build a dataset directly in Python with the client — no HuggingFace needed. Either way it caches as a lightweight preview.

2
Create a leaderboard

Bind the dataset, choose which fields are inputs vs ground truth, pick a sample subset to materialize at full resolution, and declare the prediction contract.

3
Add metrics

Attach scoring functions (Python) and visualizations. Metrics run per-sample and are pooled — averaged or otherwise aggregated — into the leaderboard score; outputs can chain into one another.

4
Submit & compare

Use benchhub-client to pull inputs, run your model, and submit typed predictions. The leaderboard ranks them and the comparison view shows every sample side by side.

Ways to bring data in

A dataset is a collection of samples and typed fields. There are two ways to create one — HuggingFace is one path among them, not the definition:

Import from HuggingFace

Tabular datasets (parquet schema) are read directly and their kinds auto-inferred. A file-tree mapper handles repos that aren't tabular — paired files, packed .npz archives, video clips — mapped field by field, with an automatic handoff when a repo isn't tabular.

(A Kaggle importer exists behind a feature flag.) See Importing.

Build it in Python

Construct a dataset directly with the client — no HuggingFace at all. Declare fields and their kinds, then add samples:

import benchhub as bh

client = bh.Client()  # reads $BENCHHUB_API_TOKEN
ds = client.create_dataset("my-data")
ds.add_field("image", kind="image", role="input")
ds.add_field("label", kind="label", role="gt")
ds.add_sample("000", image=..., label=...)
print(ds.create())

Either way the dataset caches as the preview tier and defaults to private — you publish when you're ready.

Signing in

BenchHub never stores passwords. You can sign in three ways — all create your account on first use:

  • GitHub or Google (OAuth, one click);
  • Email — enter your address and we mail you a 6-digit code to verify it (no password).

All three land you on your home dashboard. See the User Guide for accounts, quotas, and visibility.

How to read these docs

Start with this Overview to see the whole loop in one picture: bring a dataset, define a leaderboard with a typed prediction contract, submit predictions, get scored, and inspect the comparison view.

Then read Core Concepts, which builds the object model smallest to largest — kind, field (name + kind + role), sample, dataset, leaderboard, submission, metric — plus the field roles (input, ground truth, and pred — the field the submitter produces) that let one dataset be used many ways, so the rest of the docs name things you already understand.

Next, work through a first tutorial end to end: build a small dataset, create a leaderboard, submit with the client, and read the comparison view.

Only then reach for the import recipes (HuggingFace and the file-tree mapper) and the metrics and visualizations guides, pulling in the Data Types reference for the authoritative kind list — they are variations on a flow you now know, not new concepts.
