Leaderboards

Turn a dataset into a benchmark: pick a sample subset, declare the prediction contract, and bind the metrics that rank submissions.

Creating a leaderboard

Open a dataset and click Create leaderboard (the dataset page, /dataset/<id>). That opens the creation page (/dataset/<id>/create_lb) which walks you through, in one form:

  1. Name & task template. Optionally pick a task chip (e.g. classification, depth) to pre-fill sensible metrics and a prediction schema.
  2. Field roles. Mark each dataset field as input (given to submitters) or gt (the target). An image field defaults to the input when nothing else is marked.
  3. Prediction fields. The contract submitters must produce — each a name (<x>_pred) and a kind.
  4. Sample selection. Which samples to materialize (below).
  5. Metrics. Pick metrics and map each metric argument to a field.

To bind several datasets to one leaderboard, use the chooser (/create_lb) and tick the datasets. Submitting the form (POST /create_leaderboard) creates the leaderboard and kicks off materialization.

Sample selection & materialization

Datasets are stored as a cheap preview tier; a leaderboard materializes the full-resolution bytes for just the samples it scores against. You choose:

OptionMeaning
Sample capHow many samples (or all of them).
Samplinghead (first N) · random (default) · stratified (balanced across a label field).
Stratify fieldThe label field to balance on (when stratified).
SeedMakes random/stratified reproducible.

Materialization runs in the background. The leaderboard page shows a status panel:

  • pending / running — fetching the full-resolution bytes;
  • ready — with a summary of the sampling method used (e.g. “1,000 samples, stratified by label, seed 42”);
  • failed — with the error and a Retry button.

The materialized bytes count against the leaderboard owner's storage quota. You can Edit the materialization params (re-pick the subset) — but only until the leaderboard has submissions.

Re-materialization is blocked once a leaderboard has submissions. Re-picking the sample set would silently change the inputs existing submissions were scored against. The panel shows “Locked — has submissions”; delete the submissions to unlock.

The prediction contract

Each leaderboard declares exactly what a submission must contain. Edit it on the leaderboard's Edit page (/leaderboard/<id>/edit → Prediction fields tab): one row per field with a name, a kind (any data type — image, mask, depth, label, scalar, json, sequence, …), and a description.

The contract is frozen once verified submissions exist — changing a kind afterwards would re-interpret existing predictions through the wrong decoder. Delete those submissions to unlock it. Submitters can read the live contract with client.leaderboard("<owner>/<slug>").contract() — using the board's Client handle (an integer id still works too).

Binding metrics

A metric from the library is bound to a leaderboard with:

  • argument mappings — which field feeds each function argument (e.g. gt → label, pred → label_pred);
  • display name (the column header, also the name other metrics chain on);
  • pooling — mean / median / percentile / min / max across samples;
  • sort direction — higher- or lower-is-better;
  • tag filter — restrict scoring to a tagged slice of samples.

The first bound metric is the primary metric — it drives the default ranking and the score shown on cards. See Writing Metrics for authoring and chaining.

The leaderboard page

The leaderboard page (/leaderboard/<id>) shows the ranked submission table — one column per bound metric, color-graded, sortable. Per submission you can:

  • Recompute (↻) — re-run scoring without re-uploading (e.g. after adding a metric);
  • Compare — open the comparison view for a sample-by-sample look;
  • Explore samples — browse the materialized inputs/GT.
