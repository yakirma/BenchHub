Supported types

How BenchHub recognizes, stores, and serves the fields in your datasets and submissions.

BenchHub recognises a field's type from its folder name in your ZIP. The canonical convention is <type>_<field_name> — e.g. image_rgb, mask_annotation, depth_gt, scalar_score, text_caption. The prefix is authoritative; everything after the underscore is the field name BenchHub will use to refer to that field on the LB, in the comparison view, and inside metric/visualisation code. The same rules apply on the ground-truth side (a Dataset uploaded as a ZIP, or a HuggingFace dataset attachment — column names follow the same convention) and on the prediction side (a Submission uploaded via the API).

Back-compat: folders without a known prefix still get inferred from file extension + content (PIL mode + unique-value count for image-vs-mask, .npz keys for histogram-vs-depth, etc.). Mixing styles in a single ZIP works, but the explicit <type>_ prefix is always preferred — clearer to readers and avoids edge cases where an unusual photograph reads as a mask (or vice versa).

Reference table

Type Folder name File ext. Per-sample naming How it's stored / read
scalar scalar_<name> .txt {{ '.txt' }} UTF-8 text containing a single number — float or integer class index. Parsed to float for metric code.
text text_<name> .txt {{ '.txt' }} UTF-8 string. Free-form — captions, sentiment labels, short classifier outputs. Passed to metric code as a Python str.
image image_<name> .png / .jpg / .jpeg / .bmp / .tiff {{ '.' }} RGB or grayscale photograph. Read as a PIL.Image when a metric or viz asks for it. Auto-detected as mask instead when the bytes look paletted / low-color.
mask mask_<name> .png {{ '.png' }} Segmentation mask — single-channel HxW class IDs OR low-color RGB. Renders with the deterministic-hue palette; pairs naturally with IoU / Dice metrics.
depth depth_<name> .npz {{ '.npz' }} numpy .npz archive carrying the HxW float depth map (key depth). Cached as 8-bit grayscale PNG; the comparison view applies a colormap (turbo / viridis / normal-map) at render time via ?cmap=.
audio audio_<name> .wav / .mp3 / .flac {{ '.' }} Decoded via soundfile. Cached as a small waveform PNG.
histogram histogram_<name> .npz {{ '.npz' }} numpy .npz with two arrays: bins + counts. Exposes a sub_entropy_<field> convenience value to metric code.
json json_<name> .json {{ '.json' }} Arbitrary structured data, parsed to a Python dict. Bboxes, span offsets, per- sample config, etc.
metric metric_<name> .txt {{ '.txt' }} Pre-computed per-sample metric value (single float). Surfaces in the per-sample metrics chart instead of as a normal column.

Using data in metrics

Detected fields become arguments you map onto a metric function in the Leaderboard Settings. The dispatch table:

  • scalar / text → Python float / str passed by value.
  • image / mask → file path; load with PIL.Image.open(path).
  • depth → file path; load with np.load(path)['depth'].
  • histogram → pre-loaded dict with 'bins' + 'counts' arrays.
  • json → pre-loaded Python dict.

Every leaderboard page has a Submission contract widget that lists exactly which folders that LB requires — same rules as above, scoped to the metrics and visualizations attached to that LB.
