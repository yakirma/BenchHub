# templates/ — server-rendered Jinja (vanilla JS, no framework)

Big screens: `leaderboard.html`, `comparison.html`, `dataset_view.html`, `edit_leaderboard.html`. Static assets minimal (`static/css/`, `static/js/`). Theme is hardcoded in `base.html`.

## Palette (warm-cream, Cabinet retheme)
- Body bg `#fcf9f4` (warm off-white), tertiary/card bg `#f5efe2` (warm cream).
- Body/heading text `#3a2614` and `#2a1f10` (warm dark brown). Border/divider `#e8dfc8` (tan).
- Primary accent **kept violet** `#7c3aed` (brand identity). Don't swap it without an explicit ask — badges, focus rings, hover states all lean on it.
- Background radial gradients: amber `rgba(217,119,6,…)` + peach `rgba(244,114,89,…)`. If you re-tint the page bg, keep the gradient stops in the same hue family or it'll clash.

## Theme + layout conventions
- **Light-only by design.** `<html data-bs-theme="light">` is hardcoded in `base.html`. The CSS override block sets identical values for `[data-bs-theme="light"]` and `[data-bs-theme="dark"]`, but Bootstrap's own navbar vars (`--bs-navbar-color`, `--bs-tertiary-bg-rgb`) aren't covered, so any dark-mode rendering leaks white-on-white. `global_settings.theme_mode` still defaults to `'dark'` in SQLite but the template no longer reads it. Don't reintroduce a real dark mode without overriding *every* `--bs-navbar-*` and `-rgb` variant.
- **Navbar text pinned manually** (`.navbar .nav-link { color: #281950 }` etc.) as belt-and-suspenders.
- **`stretched-link` inside a sticky sidebar needs `position: relative` on the parent** — else the first card's anchor covers the whole scroll container and intercepts every later click. Bit us in the `/leaderboards` category tree.
- **Mobile pattern for long lists** (metrics, visualizations): render a `<select>` with `d-md-none`, hide the sidebar with `d-none d-md-block`. Keeps the detail pane on-screen without a Bootstrap collapse dance.
- **Dataset/leaderboard catalog cards use `.bh-card-grid`** (defined in `base.html`), NOT a Bootstrap `row row-cols-* g-3`. It's a fixed-width CSS grid (`repeat(auto-fill, minmax(12rem, 1fr))`, 2-up under 768px) so a `.home-card` is the **same size on every page** (home, landing, `/datasets`, `/leaderboards`) regardless of container width — the column count flexes, the card width doesn't. The old per-page `row-cols` counts made identical cards render at different sizes (e.g. home LB grid was 3-up while datasets were 4-up in the same column). Grid children are bare `<div>`s (no `.col`); keep `home-grid-item`/`ds-search-row` where the page's JS filter needs them. The stale `base.html` comment mentioning a "richer `.ds-card` grid" predates this — catalogs use `.home-card`.

## Comparison view (`/comparison/<lb_id>`) gotchas
- **`samples_only=1` must thread through every navigation link** (pagination, View Options form, filters) or the page collapses back into full submission-comparison mode the moment the URL drops the param.
- **`samples_only_mode` filters out only submission-needing panels** (`per_sample_metrics`, `per_source_stats`, `pred_histogram`, `viz_*`) — NOT scalar/text/json/audio columns.
- **`leaderboard.comparison_display_columns` CSV is legacy.** The renderer uses `available_display_options - hidden_comparison_display_columns`; the form persists only `hidden_*`. Don't rely on the CSV for visibility.
- **Template gates header+cell on `all_field_types.get(col_key) != 'metric'`** (was `not in ['scalar','metric']`). A new field type must not be accidentally excluded.
- Depth column header has a colormap `<select>` that rewrites every `.depth-img[data-col-key]` src in that column on change; `serve_custom_field_image` forwards `?cmap=` through its redirect to `serve_gt_viz` (Flask doesn't carry query args across `redirect()`). The depth **cmap selector is per-submission** (each pred `<select>` + `.depth-cell` carry `data-sub-id`; the change handler scopes by it so one submission's colormap doesn't recolour the others). The depth **zoom modal** (`#vis-modal`) has its own cmap `<select>` (`#modal-depth-cmap`) **+ a separate `#modal-depth-inverse` toggle button** (inverse is orthogonal to the palette, not a list entry) — both preview-only. Colormap+inverse persist **per field** for the life of one zoom session: `window._modalDepthModes` maps the thumb's `data-group-id` (e.g. `gt-disparity`, `sub-12-disparity`, sample-independent) → `{cmap, inverse}`, so a chosen mode survives up/down (same column, new sample) AND sideways (back to a field) navigation; reset on `hidden.bs.modal`. `window._applyModalDepthMode()` is the single apply path — recolours the overlay-layer stack (`#vis-modal-layers img` carrying `?cmap`, the usual case), else the static `?cmap`/`&inverse=1` image, else restyles the Plotly heatmap (`modalDepthTrace(grid, cmap, inverse)`); called on open, on every nav (so rebuilt layers re-inherit the mode), and on each control change. Server: `serve_depth_image` + `_render_depth_png` accept `&inverse=1` orthogonally (legacy `cmap=inverse` still = turbo+inverse). **`GalleryModalManager.handleKeydown` (base.html) ignores `SELECT`** (alongside INPUT/TEXTAREA) — else arrow-keying the cmap dropdown also navigated samples + rebuilt layers, eating the change ("only worked on the second try").
- **Column order** is `get_column_priority` (app.py) — sole lever, driving header + cells: sample_name → metric chart → stats → tags → GT data fields → pred data fields. Within each role block, modality order text→image→mask→depth→audio→sequence→json→hist→scalar→label. **Leaderboard viz columns sort *inside* a role block at the image-modality slot** (they render an image): a GT-only viz (arg_mappings reference no `sub_` field → `gtviz_` key, one shared column) lands in the GT block right after the raw image; a prediction viz (`viz_` key, one column per submission) lands in the pred block. Ties with the raw image field break on insertion order (data fields registered before viz). So a detection board reads `image → 📊 GT overlay → objects-json | pred-json → 📊 pred overlay`. Changing the order = edit that one function.
- **Single merged `per_source_stats` "Stats" column** (was two GT/sub cells): one table, rows = non-metric fields (scalar+label only; metrics omitted — they're in the chart), columns = GT + each submission. Gated on `has_stats_column` (`has_gt_nonmetric_stats or has_pred_nonmetric_stats`); the whole column hides when neither side has a non-metric stat.
- **Comparison shows only LB-relevant GT fields**: when `leaderboard.field_roles_json` is set, `dataset_custom_fields` is filtered to fields with role input/gt (so e.g. nyuv2's `semantic`/`instance` masks don't show on a depth-only LB). LBs with no explicit roles still show all renderable fields.
- **Sticky header + legend, page scrolls.** Site navbar is `position:fixed` (`<header class="fixed-top">`), so JS (`layoutStickyTable`) measures navbar height + legend height and sets the sticky legend (`#cmp-legend`) `top` and the `#table-container` `top`/`max-height` accordingly — the table scrolls internally so only the header row + legend pin while the controls/viz above scroll away. Forgetting the fixed-navbar offset pins the legend behind it and shoves the header up (occlusion). Top horizontal-scrollbar mirror is hidden in this mode.
- **Filter-by** (`filter_field`/`filter_value`, mirrors the dataset page): resolves to a matching sample-NAME set from GT CustomFields, applied to the real-sample query AND the HF-stub/materialised virtual-sample name list. Thread both params through pagination + the View Options form.
- **Point-track video cells** (`gtviz_`/`viz_` with `.anim`) render with **native `<video controls>`** (play/pause/scrub) wrapped in `.vis-video-wrap` + a hover `.vis-zoom-btn` — NOT autoplay-loop-click-to-modal (native controls own the click, so zoom moved to the button). The video keeps `clickable-vis` so the gallery still indexes it; the zoom button calls `bootstrap.Modal…show(siblingVideo)` (so `relatedTarget`=video, gallery indexing intact). **Row-sync**: a capture-phase `play`/`pause`/`seeked` listener mirrors any `video.rowsync-video` to the other videos in the same `<tr>` (`_rowSyncBusy` guard, cleared via `setTimeout(…,0)`). Media events don't bubble — capture phase is why the listener catches them. Modal video gets a `ZoomHandler` (stored on `_modalVideo._zoom`, one per element).
- **Side-by-side compare in the zoom modal** (`#modal-sxs`): Ctrl/⌘-click footer thumbnails or Shift+←/→ to select ≥2 gallery items → they render as `.sxs-panel`s in one row, single-viewer (`#modal-content-wrapper`) hidden. One `MultiZoom` (top-level in comparison.html) drives wheel/drag/click-to-2.5× on ALL panel `.sxs-media` with the same transform (synced). `window.visGallery` is exposed for the handlers; `window._exitVisSxs()` (called at the top of `updateVisModalContent` + on `hidden.bs.modal`) tears it down — so any plain click/arrow-nav drops back to single view. Shift+arrow handler is registered at **parse time** (window capture) so it sits ahead of `GalleryModalManager`'s per-open arrow handler and `stopImmediatePropagation()` blocks plain nav. Ctrl-click handler is **capture-phase on `#vis-gallery-footer`** so `stopPropagation()` suppresses the thumb's normal single-select. Panel videos autoplay/loop and sync play/pause/seek (`#modal-sxs video.sxs-media`). The opacity-overlay layer system (single-view blending) is untouched and coexists.
